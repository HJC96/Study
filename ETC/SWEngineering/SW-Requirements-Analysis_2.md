# 소프트웨어와 언어의 관계

## 1\. 핵심 전제: 소프트웨어는 생각이다
  * 1강의 핵심 내용인 "소프트웨어 = 생각"에서 출발.
  * **문제점**: 생각은 눈에 보이지 않고 만져지지 않기 때문에 개발과 관리가 매우 어려움.

## 2\. 생각과 언어의 관계

소프트웨어를 잘 만들기 위해 인류가 찾아낸 해답은 \*\*'언어'\*\*에 있음.

  * **철학적/과학적 결론**: "언어가 생각을 만든다."

      * 일반적인 통념: 생각이 있어서 언어가 나온다 (X)
      * **실제 증명**: **언어가 있기 때문에 생각이 발생한다 (O)**
      * *예시*: 헬렌 켈러와 같은 시청각 장애인이 언어(물, 엄마)를 습득하는 순간, 비로소 '생각'이라는 체계가 잡히고 행동이 교정됨.

  * **소프트웨어 개발에의 적용**:

      * 소프트웨어 = 생각
      * 생각 $\leftarrow$ 언어
      * **$\therefore$ 소프트웨어 $\leftarrow$ 언어**

  * **시사점**: 개발자가 사용하는 단어(자연어) 하나가 결과물에 지대한 영향을 미침.

      * *예시*: '검색기'를 만들자 vs '탐색기'를 만들자
      * 단어 하나 차이지만, 결과물(Search Engine vs Explorer)은 완전히 달라짐.

## 3\. 언어의 구조와 개발 방법론 (동사 vs 명사)

언어의 핵심 품사인 동사와 명사는 소프트웨어 개발 방법론의 두 축이 됨.

| 구분 | 절차적(함수적) 방법론 | 객체지향 방법론 |
| :--- | :--- | :--- |
| **중심 언어** | **동사 (Verb)** | **명사 (Noun)** |
| **특징** | 시간의 흐름, 순서, 움직임 강조 | 사물, 주체, 객체 강조 |
| **구현 예시** | 함수(Function) 중심 | 클래스(Class) 중심 |
| **예시 상황**<br>(코끼리 냉장고 넣기) | 1. 데려온다<br>2. 자른다<br>3. 연다<br>4. 넣는다<br>5. 닫는다 | **나**, **칼**, **냉장고**, **코끼리**<br>(객체들이 상호작용) |

## 4\. 왜 객체지향인가? (유지보수성)

  * **소프트웨어의 품질 속성**: 소프트웨어는 썩지 않지만, 요구사항은 끊임없이 변화하므로 \*\*유지보수(고쳐 쓰기)\*\*가 핵심임.
  * **변화의 빈도**:
      * **동사(행위)**: 변화가 적음.
      * **명사(사물/대상)**: 변화가 빈번함. (예: 코끼리가 호랑이로 바뀜)
  * **결론**:
      * 절차적(동사 중심)으로 짜면 대상이 바뀔 때 여러 단계의 코드를 수정해야 함.
      * 객체지향(명사 중심)으로 짜면 해당 객체(명사)만 교체하면 됨.
      * 따라서, **변화에 유연하고 유지보수가 유리한 '명사 중심(객체지향)' 설계**가 선호됨.

## 5\. 요약

  * 소프트웨어 개발 시 사용하는 **언어(단어) 선택**에 신중해야 함.
  * 말과 글이 곧 소프트웨어의 구조와 본질을 결정함.
  * 유지보수성을 고려할 때 동사보다는 명사 중심의 사고(객체지향)가 유리함.

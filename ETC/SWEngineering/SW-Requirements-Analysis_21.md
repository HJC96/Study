# 요구사항 명세(Specification)와 잘못된 사례 (Anti-Patterns)

## 1. 요구사항 명세란 무엇인가?
* **명세(Specification) = 분석의 완료:** 별도의 엑셀이나 워드 문서를 만드는 것이 필수가 아님. 앞선 실습에서 UML 도구(EA 등)를 통해 분석하고 모델링한 결과물 자체가 이미 **요구사항 명세서**임.
* **필요 시 문서화:** 특정 표준(IEEE 등)이나 인증을 위해 문서가 필요하다면, 모델링 도구의 `Publish` 기능을 이용해 문서를 생성하면 됨.

---

## 2. 핵심 개념 복습
* **요구사항(Requirement) = 관심사항(Concern) - 모호성(Ambiguity)**
* **기능 요구사항:** Input 대비 Output 값의 변화 (명확한 수식/로직).
* **비기능 요구사항:** 변화를 제약하는 품질 속성 (시나리오 + 수치).

---

## 3. 요구사항 작성 시 흔한 실수 (Anti-Patterns)

### 🚫 실수 1: "설계(Design)" 내용을 적지 마라
요구사항은 **'무엇(What)'**을 만들 것인지를 정의하는 것이지, **'어떻게(How)'** 구현할지를 적는 곳이 아님.
* **나쁜 예:** "C언어를 기반으로 하고, C++ 인터페이스를 제공한다.", "TCP/IP 소켓 통신을 사용한다."
* **문제점:** 구현 방법을 미리 강제하면, 더 좋은 기술이나 대안이 있어도 사용할 수 없게 됨. (단, 고객이 기술 스택을 명시적으로 요구한 경우, 즉 기술 자체가 컨선인 경우는 예외.)

### 🚫 실수 2: "화려한 부사(Adverb)"에 속지 마라
'빠르게', '안정적으로', '통합적으로', '손쉽게' 등 듣기 좋은 말들은 엔지니어링 관점에서는 **쓰레기(Nothing)**임.
* **나쁜 예:** "사용자가 **손쉽게** 포팅할 수 있어야 한다.", "시스템은 **안정적으로** 동작해야 한다."
* **문제점:** '손쉽다'의 기준이 모호함. 개발자는 본인 기준으로 '손쉽게' 만들고, 고객은 '어렵다'고 느껴 클레임 발생 $\rightarrow$ 재작업 무한 루프.
* **해결책:** 구체적인 시나리오와 수치로 표현해야 함 (예: "초급 개발자가 매뉴얼을 보고 1시간 이내에 포팅 완료").

### 🚫 실수 3: "동사(Verb)"를 뭉뚱그리지 마라
여러 개의 동작을 하나의 문장에 섞어 쓰면 모호해짐.
* **나쁜 예:** "사용자 정보를 **설정하고 읽어들인다**."
* **문제점:** 설정(Write)과 읽기(Read)는 엄연히 다른 기능임. 하나의 문장에 넣으면 둘 중 하나를 놓치거나 예외 처리가 부실해질 수 있음.
* **해결책:** **Atomic(원자적)**하게 분리하라.
    1.  사용자 정보를 요청한다.
    2.  사용자 정보를 수집한다.

---

## 4. 결론: "나를 위한 코딩"을 피하라
모호한 요구사항 명세서는 개발자가 **상상력**을 발휘하게 만듦.
* 개발자는 모호한 부분을 **자신이 편한 대로** 해석하고 구현함 ("이 정도면 빠르지", "이러면 손쉽지").
* 결과물은 고객의 기대와 달라지고, 결국 모든 것이 **재작업(Rework)** 대상이 됨.
* **모호성을 제거하는 분석(Analysis)**만이 프로젝트 성공의 지름길임.

# 요구사항 분석의 기초 (Context와 이해관계자)

## 1. 소프트웨어의 본질 복습 (선수 개념)
* **소프트웨어 = 생각의 일종 (계산 방법)**
    * 생각은 **언어**로부터 나오며, 언어의 핵심은 **명사(객체)**와 **동사(기능)**임.
    * 언어적 이해도가 높을수록 소프트웨어 설계와 요구사항 분석을 잘할 수 있음.
* **복잡성을 다루는 도구**
    * **모델링(Modeling):** 진짜는 아니지만 진짜를 이해하기 위해 개념을 잡는 과정.
    * **프로세스(Process):** 언어(개발)의 반복적인 수행 절차.

## 2. 요구사항 분석의 전제 조건
요구사항을 제대로 파악하기 위해서는 **Context(맥락)**와 **Stakeholder(이해관계자)** 정의가 선행되어야 함.

### A. Context (맥락)
* **정의:** 현재 시스템 상태의 **앞(Before)**과 **뒤(After)**.
* **시스템 컨텍스트:** 시스템을 중심으로 주변 환경과의 관계를 파악하는 것.

### B. Stakeholder (이해관계자)


* **정의:** 시스템에 **관심(Interest)**이나 **이해관계**를 가지고 있는 사람 또는 집단.
* **시스템의 정의:** 요소들이 모여 하나 이상의 목적을 완전히 달성하는 것.
    * *예시) 온라인 강의 시스템: 강사, 학생, 카메라, 빔프로젝터 등의 요소가 모여 '지식 전달' 목적 달성.*
* **이해관계자의 범위 (중요)**
    * 개발자들이 흔히 범하는 오류는 이해관계자를 **'최종 사용자(End User)'**로만 한정하는 것.
    * **실제 이해관계자:** 사용자뿐만 아니라 **구매자, 개발자, 디자이너, 테스터(SQA), 운영자(Operator), 유지보수 담당자, 마케터, 영업 담당자** 등 시스템 생명주기와 관련된 모든 사람.
    * **핵심:** 이해관계자를 풍부하게 찾아내야 누락 없는 요구사항 도출 가능.

## 3. 핵심 구분: 관심사항(Concern) vs 요구사항(Requirement)



| 구분 | 관심사항 (Concern/Needs) | 요구사항 (Requirement) |
| :--- | :--- | :--- |
| **정의** | 이해관계자가 시스템에 대해 바라는 것 | 시스템이 갖춰야 할 명확한 스펙 |
| **특징** | **모호성(Ambiguity)이 존재함** | **모호성이 제거된 상태** |
| **예시** | "배우기 쉬웠으면 좋겠다", "빨랐으면 좋겠다" | "메뉴 진입 2단계 이내", "응답속도 100ms 이하" |
| **주의** | **관심사항을 그대로 적는 것은 요구사항 정의가 아님.** | 분석 과정을 통해 모호함을 없애야 비로소 요구사항이 됨. |

## 4. 기능(Functional) vs 비기능(Non-Functional) 관심사항
* **기능적 관심:** 검색, 채팅, 결제 등 눈에 보이는 기능 (개발자가 주로 집중하는 영역).
* **비기능적 관심:** 유지보수성, 확장성, 보안성, 개발 비용, 개발 기간, 안정성 등.
    * 실제 이해관계자(특히 운영자, 경영진 등)는 **비기능적 요소**에 더 큰 관심을 가지는 경우가 많음.
    * 이해관계자를 폭넓게 정의하지 않으면 비기능 요구사항이 대거 누락되어 실패한 프로젝트가 될 위험이 큼.

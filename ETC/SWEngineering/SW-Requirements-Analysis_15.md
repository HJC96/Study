# 기능 요구사항 분석 실습 (Use Case Diagram)

## 1. 기능 요구사항 모델링의 종류
기능 요구사항을 분석하는 모델링 방법은 다양하며, 이번 실습 과정에서는 다음 4가지를 순차적으로 다룸.

1.  **Use Case Diagram (이번 강의 핵심):** 사용자(Actor)와 시스템 간의 상호작용 정의.
2.  **Use Case Scenario:** 유스케이스 내부의 상세 흐름 기술.
3.  **Activity Diagram:** 시나리오를 활동(Activity) 흐름으로 도식화.
4.  **Sequence Diagram:** 시스템 내부 객체 간의 순차적 메시지 흐름 파악.

---

## 2. Use Case Model의 특징
* **Actor 중심:** 사용자(Actor)와 시스템의 상호작용을 통해 요구사항을 분석.
* **기능(Functional) 중심:** **비기능 요구사항(품질, 제약 등)은 표현하지 않음.**
    * *비기능까지 포함하면 복잡도가 지나치게 증가하므로 권장하지 않음.*

## 3. 주요 구성 요소 (Notation)

| 요소 | 설명 | 표기법 |
| :--- | :--- | :--- |
| **Actor** | 시스템을 사용하는 사람 또는 외부 시스템.<br>(예: 교수, 학생) | 졸라맨(Stick figure) 모양 |
| **System Boundary** | 시스템의 경계. <br>**내부:** Use Case (기능)<br>**외부:** Actor (사용자) | 사각형 박스 |
| **Use Case** | 시스템이 제공하는 개별 기능.<br>**반드시 '동사'로 끝나야 함.**<br>(예: 수강신청'하다', 성적입력'하다') | 타원형 |

---

## 4. 유스케이스 간의 관계 (Relationships)
가장 혼동하기 쉬운 **Include**와 **Extend**의 차이를 명확히 구분해야 함.

| 구분 | Include (포함) | Extend (확장) |
| :--- | :--- | :--- |
| **개념** | **필수(Mandatory)** 실행 | **선택(Optional)** 실행 |
| **설명** | A가 실행되면 **반드시** B도 실행됨. | A가 실행될 때, **특정 조건**에서만 B가 실행됨. |
| **화살표 방향** | **Base $\rightarrow$ Included**<br>(원래 기능에서 포함된 기능으로) | **Extension $\rightarrow$ Base**<br>(확장 기능에서 원래 기능으로) |
| **예시** | '강좌 관리' 시 '강좌 개설'은 필수 | '성적 조회' 시 '성적 수정'은 선택적 상황 |

---

## 5. 실습 예제: 수강신청 시스템
* **도구:** Enterprise Architect (EA) 사용 (다른 UML 도구도 무관).
* **구성:**
    1.  **시스템 바운더리:** '수강신청 시스템' 설정.
    2.  **Actor 도출:**
        * **교수:** 강좌 관리, 성적 처리.
        * **학생:** 수강 신청, 성적 조회.
    3.  **Use Case 도출 (동사형 명명):**
        * 강좌를 개설하다 (Include/Extend 관계 설정).
        * 수강을 신청하다.
        * 성적을 입력하다 / 조회하다.

## 6. 핵심 가이드
* **명명 규칙:** Use Case 이름은 **명사**가 아닌 **동사**로 끝내야 다이어그램이 자연스러움. (예: `강좌 관리` (X) $\rightarrow$ `강좌를 관리하다` (O))
* **위치:** Actor는 반드시 시스템 바운더리 **밖에**, Use Case는 **안에** 위치해야 함.
* **분해:** 너무 복잡한 기능은 Include/Extend를 사용해 기능적으로 분해(Decomposition)하여 중복을 피하고 가독성을 높임.

# 9강 요약: Agile vs UP, 요구사항 분석의 차이

## 1. Agile에 대한 오해와 진실
* **오해**: Agile은 요구사항 분석을 하지 않는다?
* **진실**: 요구사항 분석은 필수.
    * 단, 무거운 문서 작업이 아님.
    * **빠르고 반복적**으로 수행됨.
    * **워킹 소프트웨어(Working Software)**를 만드는 과정 자체에 분석이 포함됨.

## 2. UP(Unified Process) vs Agile 비교

| 구분 | UP (Unified Process) | Agile (Scrum 등) |
| :--- | :--- | :--- |
| **중심 가치** | **계획(Plan)** 중심 | **고객(Customer)** 중심 |
| **고정 요소** | **일(Scope)** 고정 <br> (일을 완수하기 위해 시간을 조정) | **시간(Time)** 고정 <br> (정해진 시간 내에 할 수 있는 일을 조정) |
| **질문 방식** | "이 기능 구현에 **얼마나 걸리는가?**" | "**이번 주까지** 무엇을 구현할 수 있는가?" |
| **핵심 산출물** | **모델(Model)** 중심 <br> (UML, 설계 문서) | **워킹 소프트웨어** 중심 <br> (실제 작동하는 코드, 데모) |
| **분석 방식** | 단계별(Inception 등) 모델링 후 구현 | Backlog(데모 시나리오) 작성 후 즉시 구현 |

## 3. 직관적 비유: 2차 방정식 풀이
* **UP 방식 (근의 공식 대입)**
    * 공식을 사용해 단계별로 논리적 전개 (분석 → 설계 → 구현).
    * 체계적이고 정석적인 접근.
* **Agile 방식 (숫자 대입법)**
    * 정답 범위를 유추하고 숫자를 넣어봄(구현).
    * 결과가 맞는지 확인(데모/피드백) 후 수정.
    * **"일단 만들어서 돌려보고 아니면 고친다"**는 빠른 피드백 루프가 핵심.

## 4. 결론: 소프트웨어 개발의 본질
* **소프트웨어 = 언어(Language) + 반복(Iteration)**
* **반복의 목적 차이**
    * **UP**: 계획된 **일(Scope)**을 완벽히 수행하기 위한 반복.
    * **Agile**: 고정된 **시간(Timeboxing)** 내에 가치를 검증하기 위한 반복.

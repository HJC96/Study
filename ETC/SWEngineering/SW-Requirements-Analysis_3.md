# 소프트웨어 모델링의 본질

## 1\. 소프트웨어 개발의 양대 축

소프트웨어를 잘 만들기 위해 필요한 개발 방법론은 크게 두 가지 요소로 구성됨.

1.  **모델링(Modeling)**: 무엇을 만들 것인가 (구조와 형상).
2.  **프로세스(Process)**: 어떤 순서와 절차로 만들 것인가.
      * *이번 강의의 핵심 주제는 **'모델링'**.*

## 2\. 모델(Model)이란 무엇인가?

  * **정의**: **"진짜가 아닌데, 진짜 같이 보이는 것."**
  * **핵심 속성**:
      * **대표성 (Representation)**: 대상을 대표함.
      * **추상화 (Abstraction)**: 불필요한 부분은 생략하고 특징을 강조함.
  * **일상의 예시**:
      * **아파트 모델하우스**: 진짜 집은 아니지만, 거주 시의 가치(전망, 편의성 등)를 미리 체험하게 해줌.
      * **패션 모델**: 내가 아니지만, 옷을 입었을 때의 느낌(핏, 스타일)을 상상하게 해줌.
  * **목적**: 진짜를 바로 만들기에는 비용과 시간이 많이 들거나(건축), 진짜의 실체가 모호할 때(소프트웨어) 미리 검증하고 가치를 판단하기 위함.

## 3\. 소프트웨어 모델링의 특수성

  * **정형 모델 vs 비정형 모델**:
      * **정형 모델(Geometric Model)**: 자동차, 건축물처럼 눈에 보이는 형태가 있음 (그리기 쉬움).
      * **비정형 모델(Non-geometric Model)**: 소프트웨어, 음악, 수학처럼 형태가 없음 (그리기 어려움).
  * **소프트웨어의 난제**: '생각'을 닮아 형태가 없기 때문에 모델링(진짜처럼 보이게 만드는 것)이 매우 어려움.

## 4\. 요구사항 분석 모델의 목표

  * **소프트웨어의 '진짜(Real)'는 무엇인가?**:
      * 컴퓨터 위에서 실행되어 가치를 제공하는 **소스 코드(Source Code)** 그 자체.
      * (컴파일과 실행은 자동화되어 있으므로, 개발자가 만드는 본질은 소스 코드임).
  * **좋은 요구사항 분석 모델이란?**:
    \*

```
* 모델(요구사항 정의서)을 보는 순간 **소스 코드가 눈에 보여야 함.**
* 사용자 입장에서 쓴 **명사**와 **동사**가 개발 단계에서 **클래스**와 **함수**로 매핑되어 보여야 함.
* 만약 모델을 봤는데 코드가 연상되지 않는다면, 그것은 '진짜 같이 보이지 않는' 실패한 모델임.
```

## 5\. 결론 및 해법

  * 형태가 없는 소프트웨어를 모델링하는 유일한 방법은 \*\*'언어 모델링'\*\*임.
  * 언어(요구사항 텍스트)를 분석하여 그 구조를 파악하고, 이를 소스 코드의 구조(클래스, 함수)로 연결하는 것이 핵심.
  * **작명(Naming)**: 요구사항 분석 단계에서 소스 코드의 변수명, 클래스명의 50% 이상이 결정됨.

# 소프트웨어 개발 프로세스와 SDLC (강의 요약 - 5강)

## 1\. 소프트웨어 개발 프로세스란?

  * **정의**: 사용자의 요청부터 시스템 개발 완료까지 **해야 하는 일의 순서**를 정의한 것.
  * **소프트웨어 프로세스의 난제**:
      * 소프트웨어는 눈에 보이지 않아 '목적지'가 명확하지 않음.
      * 과거(20년 전)에 비해 규모(10만 라인 $\rightarrow$ 1,000만 라인), 복잡도, 다양성이 폭발적으로 증가함.
      * 사용자의 요구(시작점)와 실제 제품(도착점) 사이의 거리가 매우 멀어짐(개울물 $\rightarrow$ 한강).
  * **모델링과의 관계**: 모델링은 프로세스라는 긴 여정을 건너기 위한 **디딤돌** 역할을 함.

## 2\. SDLC (Software Development Life Cycle)

소프트웨어 개발 생명주기에는 항상 빠지지 않는 4가지 핵심 단계가 존재함. 이는 인간의 사고 과정(문제 해결 과정)과도 동일함.

| 단계 | 정의 | 비고 (일상의 예시: 화난 여자친구 달래기) |
| :--- | :--- | :--- |
| **분석 (Analysis)** | 주어진 데이터를 **나누어(分)** 목적을 달성하는 것. | 왜 화가 났는지 과거 데이터와 상황을 나름대로 분석함. |
| **설계 (Design)** | 구현하는 방법을 미리 정의하는 것. | 밥을 사줄지, 꽃을 줄지 해결책을 미리 구상함. |
| **구현 (Implementation)** | 현실 세계에 없는 것을 있게 만드는 것. | 실제로 꽃을 사서 전달함. |
| **테스트 (Test)** | 성공과 실패를 구분하는 행위. | 상대방의 표정을 살피며 화가 풀렸는지 확인함. |

## 4\. 워터폴(Waterfall) 모델 (폭포수 모델)

  * **개념**: 분석 $\rightarrow$ 설계 $\rightarrow$ 구현 $\rightarrow$ 테스트를 시간 순서대로 한 번씩만 진행하는 방식.
  * **특징**: 물이 떨어지면 다시 올라가지 못하듯, 전 단계로 되돌아가는 것이 어렵거나 불가능함을 전제함.

  * **타 공학(건축, 자동차)과의 차이**:
      * 건축/자동차는 **정형 모델**이므로 설계도만 봐도 결과물의 실체가 명확히 보임 (실패 확률 낮음).
      * 소프트웨어는 **비정형 모델**이므로 설계도(모델)만으로는 결과물을 확신할 수 없음.
  * **위험성**:
      * "분석은 설계해 봐야 알고, 설계는 구현해 봐야 안다."
      * 다 만들고 나서(구현 후) 분석이 잘못되었음을 깨닫게 되면, 되돌리기엔 비용과 시간이 너무 많이 소요됨.
      * 따라서 소프트웨어 개발에서 순차적 워터폴 방식은 **실패할 확률이 매우 높음**.

## 5\. 대안 및 향후 주제

  * **스파이럴(Spiral) / 진화적(Evolutionary) 모델**:
      * 워터폴의 단점을 보완하기 위해 분석\~테스트 과정을 짧은 주기로 반복 수행함.
      * 

```
* 하지만 이 역시 작은 워터폴의 반복일 뿐, 근본적인 위험성이 완전히 해소되지는 않음.
```

### 1. 기능 요구사항 복습 (Function)
* **정의:** Input 대비 Output 값의 변화.
* **핵심:** 시스템의 존재 이유. (예: 자동차가 이동하지 않으면 고철 덩어리)
* **작성법:** 반드시 **Input → Output**의 변화가 명시되어야 함.

### 2. 비기능 요구사항 (Non-Functional Requirement)
비기능 요구사항은 크게 **제약사항(Constraint)**과 **품질(Quality)**로 나뉨.
* **제약사항:** 비용, 기간, 법률 등. (명확하므로 분석 대상에서 제외되는 경우가 많음)
* **품질(Quality):** 모호성이 가장 많은 영역이자 **분석의 핵심 대상**.

### 3. 품질(Quality)의 본질: '부사(Adverb)'
기능이 시스템의 움직임(동사)이라면, 품질은 그 움직임을 **제약/수식하는 요소(부사)**.

| 구분 | 기능 (Function) | 품질 (Quality) |
| :--- | :--- | :--- |
| **품사 비유** | **동사 (Verb)** | **부사 (Adverb)** |
| **예시** | "이동한다" | "**빠르게**, **안전하게**, **편리하게**" 이동한다 |
| **테스트** | **1회** 성공 시 통과 (On/Off) | **N번** 반복 테스트 필요 (통계/확률적 접근) |
| **실패 시** | 시스템 존재 가치 상실 (Dead) | 시스템 가치 하락 (Bad Quality) |



### 4. 5대 핵심 품질 속성
1.  **성능 (Performance):** 얼마나 빠른가? (Throughput, Latency)
2.  **안정성 (Stability/Reliability):** 죽지 않고 계속 도는가?
3.  **유지보수성 (Maintainability):** 수정 및 확장이 쉬운가?
4.  **사용성 (Usability):** 쓰기 편한가?
5.  **보안 (Security):** 안전한가?

### 5. 기능성(Functionality)의 함정
품질을 높이기 위해 **추가적인 기능**을 도입하는 경우입니다. 이를 **기능성(Functionality)**이라고 부릅니다.

* **예시 (자동차 오토 와이퍼):**
    * '편리하게(품질)' 쓰기 위해 '자동으로 닦는 기능'을 추가함.
    * 오토 와이퍼가 고장 나도 자동차의 본질(이동)은 유지됨 → **본질적 기능이 아님.**
* **해결 가능성:**
    * **사용성, 보안:** 주로 **기능 추가**로 해결 가능 (예: UI 개선, 로그인 기능).
    * **성능, 안정성, 유지보수성:** 단순히 기능을 추가해서 해결하기 어려움. **아키텍처 설계**가 필요함.

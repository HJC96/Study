# 요구사항 분석의 핵심 기법 (MECE와 직교성)

## 1. 분석의 전제와 목적
* **문제 상황:** 이해관계자의 언어(관심사항)는 매우 모호함 (예: "불편해요", "안정적으로 해주세요").
* **분석의 필요성:** 컴퓨터는 0과 1의 세계이므로 모호성을 허용하지 않음. 모호한 상태로 코딩할 수 없음.
* **분석의 정의:** 주어진 데이터(관심사항)를 **나누어(Divide)** 모호성을 제거하는 과정.

## 2. 핵심 분석 기법 ①: MECE (미시)
데이터를 체계적으로 나누는 방법론.

* **개념:** **M**utually **E**xclusive, **C**ollectively **E**xhaustive
    * **Mutually Exclusive (상호 배제):** 나누어진 집합 간에 **교집합(중복)**이 없어야 함. (개발자의 `Mutex` 개념과 유사)
    * **Collectively Exhaustive (전체 포괄):** 나누어진 집합을 합치면 **전체 집합(누락 없음)**이 되어야 함.
* **비유: 스무고개 놀이**
    * 질문: "생물인가? (Yes/No)" → 세상의 모든 존재를 생물/무생물로 나눔 (MECE).
    * 효과: 질문 하나로 모호성의 절반이 사라짐. 20번의 질문으로 세상의 모든 사물을 특정 가능.
    * 잘못된 질문: "내 생각했지?" (Yes/No로 모호성이 제거되지 않음).

## 3. 핵심 분석 기법 ②: Orthogonality (직교성)
나누어진 요소들이 서로 독립적인 관계를 유지하는 것.

* **개념:** 수학적 그래프의 X축과 Y축처럼 **90도(수직)**를 이루는 관계.
    * 한 축(X)의 변화가 다른 축(Y)에 전혀 영향을 주지 않는 상태 ($y=1$ 그래프 처럼).
* **소프트웨어에서의 의미:**
    * **Decoupling (결합도 감소):** A를 수정해도 B가 망가지지 않음.
    * **효과:** 설계가 깔끔해지고, 소스 코드 양이 약 30% 감소하며, 아키텍처 수립이 용이해짐.
    * **도구:** 엑셀(표/Matrix)은 행과 열이 직교하므로 훌륭한 분석 도구임.

## 4. 요구사항의 분류: 기능 vs 비기능
소프트웨어 요구사항은 **MECE**하고 **Orthogonal**하게 다음 두 가지로 분류됨.

| 분류 | 기능 요구사항 (Functional) | 비기능 요구사항 (Non-Functional) |
| :--- | :--- | :--- |
| **정의** | 시스템이 **'무엇을 하는가'** (동작) | 시스템이 **'어떻게 동작하는가'** (품질/제약) |
| **관계** | **Orthogonal (직교)**: 기능이 추가된다고 품질 기준이 바뀌지 않고, 품질 기준을 높인다고 기능이 바뀌지 않음. |
| **세부 구성** | - | 1. **제약사항(Constraint):** 비용, 기간 등 (명확하므로 분석 불필요)<br>2. **품질(Quality):** 속도, 안정성 등 (모호하므로 **집중 분석 대상**) |

### 결론
* 요구사항 분석 시 무조건 **기능**과 **비기능**으로 나누어라.
* 이 분류는 MECE(중복 없고 누락 없음)하며, Orthogonal(서로 독립적)하므로 모호성을 제거하는 가장 확실한 첫걸음이다.

# 16강 요약: 유스케이스 시나리오 (Use Case Scenario) 작성법

## 1. 시나리오 작성의 목적
* **목차 채우기:** 유스케이스 다이어그램이 시스템의 구조(목차)라면, 시나리오는 그 내부의 **구체적인 내용**임.
* **모호성 제거:** 수학적 증명처럼 완벽하게 쓰기는 어렵고, 줄글로 쓰면 모호해짐. 따라서 **'특정한 상황(Scenario)'**을 가정하여 소설처럼 구체적으로 묘사함으로써 모호성을 없앰.
    * *예시: "현금을 인출한다" (X) $\rightarrow$ "사용자가 카드를 넣고, 비밀번호를 입력하고, 시스템이 잔액을 확인하고..." (O)*

## 2. 시나리오의 핵심 구성 요소


[Image of Use Case Scenario template structure]


시나리오는 크게 **개요, 흐름(Flow), 조건**으로 구성됨.

### A. 개요 (Overview)
* 유스케이스에 대한 간략한 정의 및 설명.

### B. 흐름 (Flow) - **핵심**
시스템의 동작 순서를 기술하며, 다음 3가지로 구분됨.

1.  **기본 흐름 (Basic Flow):**
    * 가장 전형적이고 이상적인 성공 케이스 (Happy Path).
    * *예: 수강신청 성공.*
2.  **대안 흐름 (Alternative Flow):**
    * 기본 흐름에서 벗어나지만, 비즈니스 로직상 **있을 수 있는** 정상적인 분기.
    * *예: 수강신청 기간이 아님, 수강 인원 마감 등.*
3.  **예외 흐름 (Exception Flow):**
    * 시스템 에러, DB 다운 등 **있어서는 안 되지만 발생한** 상황.
    * *예: 검색 중 DB 연결 실패.*
    * **Tip:** 예외 흐름을 많이 작성할수록 시스템의 안정성(Stability)이 높아짐. (권장 비율: 기본 1 : 대안 2 : 예외 3)

### C. 조건 (Conditions)
* **사전 조건 (Pre-condition):** 유스케이스 시작 전 만족되어야 할 상태 (예: 로그인 됨).
* **사후 조건 (Post-condition):** 유스케이스 종료 후 시스템 상태 (예: 수강신청 완료됨).

## 3. 시나리오 작성의 기술 (Writing Rules)

### ① User vs System (Ping-Pong 방식)
기능(Function)의 정의인 **Input(사용자) 대비 Output(시스템)의 변화**를 명확히 기술해야 함.
* **Bad:** "수강신청을 한다." (누가 무엇을 하는지 모호함)
* **Good:**
    * **1.1 (User):** 사용자는 '조회' 버튼을 클릭한다. (Input)
    * **1.2 (System):** 시스템은 개설 강좌 목록을 화면에 출력한다. (Output)

### ② 구체적인 묘사 (Specifics)
추상적인 표현을 피하고, 마치 눈앞에서 동작하는 것처럼 구체적으로 기술해야 함.
* **Bad:** "에러 메시지를 출력한다."
* **Good:** "시스템은 **'시스템 점검 중이니 잠시 후 다시 시도해주세요'**라는 메시지를 **팝업 창**으로 띄운다."
    * *구체적일수록 UI/UX 설계와 개발이 명확해짐.*

### ③ 흐름 번호 매기기
* 기본 흐름에서 분기되는 지점을 명확히 표시 (예: 기본 흐름 3번 단계에서 대안 흐름 A1으로 분기).

---

## 4. 실습 도구 (Enterprise Architect)
* 도구 내 'Scenario' 탭을 활용하여 텍스트 기반으로 관리 가능.
* 별도의 워드 문서를 만들지 않고 모델링 도구 안에서 통합 관리하는 것을 권장.

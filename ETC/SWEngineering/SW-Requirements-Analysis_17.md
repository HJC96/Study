# 시나리오의 시각화 (Activity Diagram)

## 1. 액티비티 다이어그램의 목적
* **텍스트의 한계 보완:** 유스케이스 시나리오(텍스트)가 '소설'이라면, 액티비티 다이어그램은 **'만화'**와 같음.
    * 텍스트는 구체적인 정황 묘사에 좋지만, 전체적인 로직 흐름을 한눈에 파악하기는 어려움.
    * 이를 보완하기 위해 그래픽 노테이션(UML)을 사용하여 순차적 행동과 분기를 시각화함.
* **로직의 검증:** 시나리오에서 작성한 기본 흐름, 대안 흐름, 예외 흐름이 논리적으로 말이 되는지 그림으로 그리며 검증.

---

## 2. 주요 구성 요소 (Notation)


| 요소 | 설명 | 비고 |
| :--- | :--- | :--- |
| **Initial Node** | 시작점 | 검은 동그라미 (●) |
| **Activity** | 수행하는 행위/동작 | 둥근 사각형 |
| **Final Node** | 종료점 | 이중 동그라미 (◉) |
| **Decision** | 분기점 (조건 판단) | **마름모 (◇)** <br> Flowchart의 분기와 유사 |
| **Control Flow** | 흐름선 | 화살표 |
| **Swimlane (Partition)** | **구획면** <br> 행위의 주체(Actor/System)를 구분 | 수영장 레인처럼 생김 |

---

## 3. 시나리오 $\rightarrow$ 다이어그램 변환 프로세스

### ① Swimlane(구획면) 설정 **(중요)**
* 기능 요구사항의 핵심인 **Input(사용자) 대비 Output(시스템)** 구조를 표현하기 위해 공간을 나눔.
* **좌측:** 사용자 (User/Student)
* **우측:** 시스템 (System)
* *효과: 사용자가 요청(Input)하면 시스템이 반응(Output)하는 'Ping-Pong' 흐름이 명확해짐.*

### ② 흐름(Flow) 매핑
* **기본 흐름 (Basic Flow):** 위에서 아래로 쭉 뻗어 나가는 메인 줄기(Main Stream)로 배치.
* **분기 처리 (Decision):**
    * 시나리오의 **대안 흐름(Alternative)**과 **예외 흐름(Exception)**은 마름모(Decision)를 통해 옆으로 빠지도록 그림.
    * *예: "수강신청 기간인가?" $\rightarrow$ No $\rightarrow$ 에러 메시지 출력 (예외 흐름)*

### ③ 상세 작성 팁
* **Activity 명명:** 시나리오에 있는 문장(예: "조회 버튼을 클릭한다")을 그대로 옮겨 적음.
* **조건 명시:** 화살표(Control Flow) 위에 `[Yes]`, `[No]` 또는 구체적인 조건(예: `[목록이 비어있음]`)을 명시.

---

## 4. 실습 도구 활용 (Enterprise Architect)
* 유스케이스 하위에 `New Child Diagram` $\rightarrow$ `Activity`를 생성하여 관리.
* 텍스트로 된 시나리오를 복사해두고, 하나씩 블록으로 옮기면서 로직을 연결하는 방식 권장.
* **레이아웃:** 좌우(Horizontal) 배치보다 **상하(Vertical)** 배치가 흐름을 파악하기에 더 직관적임.

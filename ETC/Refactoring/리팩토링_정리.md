기본 기술
---
여기서는 지금까지 배운 리팩토링들을 “냄새(code smell)” 기준이 아니라, 리팩토링 기술 카탈로그 기준으로 다시 묶어서 설명하고 있다. 그중에서도 가장 기본이 되는, 그리고 가장 자주 쓰이는 리팩토링 기술들을 먼저 정리한다는 내용이다. 이 기본 기술들은 다른 리팩토링을 할 때 기초 재료처럼 계속 사용된다.

가장 먼저 나오는 것이 함수 추출하기(Extract Function) 와 함수 인라인하기(Inline Function) 이다.
이 둘은 서로 반대 개념이다. 어떤 코드 일부를 따로 함수로 빼내면 함수 추출하기이고, 반대로 함수로 되어 있던 내용을 다시 본문에 넣으면 인라인하기다. 상황에 따라 가독성이 좋아지는 쪽을 선택하면 된다.

이와 같은 개념은 함수뿐 아니라 변수에도 동일하게 적용된다.
어떤 계산식이 코드에 그대로 들어가 있으면 인라인 상태이고, 그 계산식을 변수에 담아서 이름을 붙이면 변수 추출하기가 된다. 계산식의 의미를 이름으로 설명할 수 있으면 변수로 추출하는 것이 좋고, 한 번만 쓰고 굳이 이름이 필요 없으면 인라인하는 것이 더 낫다. 즉, 이해하기 쉽게 만드는 방향으로 추출과 인라인을 상황에 맞게 선택하면 된다.

그 다음으로 중요한 것이 함수 선언 변경하기이다.
이건 매우 자주 사용하는 리팩토링으로, 예를 들면:
	•	함수 이름 변경
	•	매개변수 추가/삭제
	•	매개변수 순서 변경
	•	반환 타입 변경
같은 것들이 여기에 해당한다.

또 비슷한 맥락으로 변수 이름 바꾸기도 있다.
주석을 많이 쓰는 것보다, 이름을 잘 짓는 것이 더 좋은 코드라는 의미다. 주석으로 설명할 바에는 함수 이름과 변수 이름을 더 명확하게 바꾸는 것이 낫다.

그리고 다른 기본 기술들로는 다음과 같은 것들이 있다.
	•	변수 캡슐화하기: 변수를 직접 접근하지 못하게 하고 getter/setter 등을 통해 접근하게 만드는 것
	•	매개변수 객체 만들기: 매개변수가 너무 많을 때 하나의 객체로 묶는 것
	•	여러 함수를 클래스로 묶기: 관련된 데이터와 함수들을 하나의 클래스로 모으는 것
	•	변환 함수를 묶기: 데이터를 변환하는 함수들을 한 곳에 모으는 것
	•	단계 쪼개기: 하나의 함수가 여러 일을 하면, 단계별로 나누는 것

캡슐화
---
캡슐화란 어떤 모듈 내부에 있는 데이터나 구현 세부사항을 외부에 드러내지 않는 것이다. 외부에서는 내부가 어떻게 생겼는지 몰라도, 필요한 기능만 사용할 수 있도록 만드는 것이 캡슐화다.

가장 대표적인 예가 레코드 캡슐화하기이다. 예를 들어 public 필드로 데이터를 아무나 직접 바꿀 수 있게 하면 캡슐화가 깨진다. 그래서 필드를 직접 접근하지 못하게 하고, getter/setter 같은 메서드를 통해서만 접근하게 만드는 것이 레코드 캡슐화다.

비슷한 개념으로 컬렉션 캡슐화하기도 있다.
클래스 내부에 List 같은 컬렉션이 있을 때, 그 컬렉션을 그대로 외부에 반환해버리면 외부에서 마음대로 add, remove를 할 수 있다. 그러면 그 클래스는 자기 데이터를 통제하지 못하게 된다. 그래서 방법은 보통 다음 두 가지다.
	•	컬렉션의 복사본을 반환한다.
	•	컬렉션을 직접 건드리게 하지 말고,
addItem(), removeItem() 같은 메서드만 제공한다.

즉, 컬렉션 자체를 주지 말고 컬렉션을 다루는 기능만 제공하는 것이 핵심이다.

그리고 캡슐화는 단순히 필드만 숨기는 것이 아니라, 클래스와 클래스 사이의 관계도 숨길 수 있다. 여기서 나오는 리팩토링이 Hide Delegate(위임 숨기기) 이다.

예를 들어 이런 코드가 있으면:

employee.getDepartment().getManager()

이 코드를 보면 우리는 다음 구조를 알게 된다.
	•	employee 안에 department가 있고
	•	department 안에 manager가 있다

즉, 객체 구조가 외부에 다 노출된 것이다.
그래서 이것을 캡슐화하면 이렇게 바꾼다.

employee.getManager()

그러면 외부에서는 employee가 내부적으로 department를 거치는지, 다른 방법으로 manager를 찾는지 전혀 몰라도 된다.
이게 객체 구조(의존 관계)를 숨기는 캡슐화다.

그런데 반대로, 너무 많이 숨기면 문제가 생긴다.
모든 호출이 어떤 중간 객체를 계속 거쳐야 해서 코드가 쓸데없이 길어지고 복잡해질 수 있다. 이때 사용하는 리팩토링이 Remove Middleman(중개자 제거) 이다. 즉, 불필요한 위임 메서드를 제거하고 직접 접근하게 하는 것이다.

그리고 마지막으로 중요한 개념이 하나 나오는데,
캡슐화가 잘 되어 있으면 알고리즘을 바꾸기 쉬워진다는 것이다.

예를 들어 어떤 월세 계산 로직, 날짜 계산 로직이 함수 안에 캡슐화되어 있으면, 내부 계산식을 완전히 다른 알고리즘으로 바꿔도 외부 코드는 하나도 수정할 필요가 없다.
이게 캡슐화의 가장 큰 장점이다.

기능 옮기기
---
여기서 말하는 “기능”은
	•	함수
	•	필드
	•	코드 몇 줄 (statement)
이런 것들을 의미한다.

1. 가장 중요한 것: 함수 옮기기 / 필드 옮기기

이게 기능 옮기기의 핵심이다.

어떤 함수가 A 클래스보다 B 클래스의 데이터를 더 많이 사용한다면,
그 함수는 A가 아니라 B로 옮기는 게 맞다.

즉 기준은 항상 이것이다:

이 코드는 어떤 데이터와 더 밀접한가?
그 데이터가 있는 곳으로 옮긴다.

이게 객체지향에서 말하는
“데이터와 행동을 같은 곳에 둬라” 원칙이다.

2. 문장을 함수로 옮기기 vs 문장을 호출한 곳으로 옮기기

이 둘은 서로 반대다.

문장을 함수로 옮기기

어떤 코드가 함수 호출할 때마다 항상 같이 실행된다면,
그 코드는 함수 안에 들어가는 게 맞다.

log();
calculatePrice();

이게 항상 같이 실행된다면 → log()를 calculatePrice() 안으로 넣는다.

즉,

항상 같이 실행되는 코드는 함수 안으로 들어간다.

문장을 호출한 곳으로 옮기기

반대로 어떤 코드가 특정 상황에서만 실행된다면,
그 코드는 함수 안에 있으면 안 되고 호출한 쪽에 있어야 한다.

즉,

항상 실행되는 게 아니라면 함수 밖으로 빼야 한다.

이건 보통 함수 추출을 잘못했을 때 다시 되돌리는 느낌이다.

3. 인라인 코드를 함수 호출로 바꾸기

이건 사실상 Extract Method(함수 추출) 이랑 거의 같은 말이다.

언제 쓰냐면:
	•	코드가 여러 곳에서 중복된다
	•	그 코드에 이름을 붙이면 이해가 쉬워진다
	•	재사용 가능하다

그러면 함수로 빼서 호출로 바꾼다.

4. 문장 슬라이드하기

이건 진짜 단순한 리팩토링인데 중요하다.

코드를 위아래로 옮겨서:
	•	관련 있는 코드끼리 모으고
	•	변수 선언과 사용하는 위치를 가깝게 하고
	•	그러면 함수 추출하기가 쉬워지고
	•	중복 코드도 발견하기 쉬워진다

즉, 이건 리팩토링을 하기 위한 준비 작업 같은 느낌이다.

5. 반복문 쪼개기

반복문 안에서 여러 가지 일을 하고 있으면:

for (...) {
    총합 계산
    평균 계산
    로그 출력
}

이렇게 하지 말고

for (...) { 총합 계산 }
for (...) { 평균 계산 }
for (...) { 로그 출력 }

이렇게 나누면,
	•	각각을 함수로 추출할 수 있고
	•	그 함수들을 적절한 클래스로 옮길 수도 있다

그래서 결국 이것도 기능을 적절한 위치로 옮기기 위한 과정이다.

6. 반복문 → 파이프라인 (Stream, LINQ 등)

for문을 없애고

list.stream()
    .filter(...)
    .map(...)
    .sum();

이런 식으로 바꾸는 것.
이건 기능을 옮긴다기보다는 코드를 더 선언적으로 바꾸는 리팩토링에 가깝다.

7. 죽은 코드 제거

이건 말 그대로:
	•	사용 안 하는 코드
	•	절대 실행 안 되는 코드
	•	옛날 코드

그냥 삭제하는 것.

이것도 중요하다. 죽은 코드가 많으면 구조 이해가 어려워진다.

데이터 조직화
---
이번 카탈로그는 데이터 조직화 (Organizing Data), 즉 변수와 필드를 어떻게 다룰 것인가에 대한 리팩토링이다. 핵심은 한 문장으로 정리할 수 있다.

데이터는 의미가 명확해야 하고, 꼭 필요한 데이터만 존재해야 한다.

여기서 말하는 데이터는 주로 변수와 필드를 의미한다.

1. 하나의 변수는 하나의 의미만 가져야 한다

예를 들어 temp 같은 변수를 만들어서:
	•	처음에는 기본값 저장
	•	나중에는 추가 비용 저장
	•	마지막에는 최종 결과 저장

이렇게 쓰면 이 변수는 의미가 계속 바뀌는 변수다. 이런 변수는 읽는 사람이 이해할 수 없다. 그래서 이런 경우는 변수를 나눠야 한다.

double basePrice;
double tax;
double totalPrice;

이렇게 변수 하나 = 의미 하나가 되도록 만드는 것이 데이터 조직화의 첫 번째 원칙이다.

2. 필드 이름 바꾸기

이건 변수 이름 바꾸기와 같은 개념이다.
이름이 곧 문서이기 때문에, 이름이 명확하면 주석이 줄어든다.

3. 파생 변수를 질의 함수로 바꾸기 (Replace Derived Variable with Query)

이게 중요한 개념이다.

예를 들어:

int quantity;
int itemPrice;
int totalPrice; // quantity * itemPrice

여기서 totalPrice는 사실 저장할 필요가 없다.
왜냐하면 quantity * itemPrice로 계산할 수 있는 값이기 때문이다.

그래서 이렇게 바꾼다:

int getTotalPrice() {
    return quantity * itemPrice;
}

즉,

계산으로 구할 수 있는 값은 변수로 저장하지 말고 함수로 계산해서 사용하자.

이걸 하는 이유는:
	•	데이터 중복 제거
	•	값 불일치 버그 방지
	•	상태(state)를 줄이기 위해서

이건 객체지향에서 굉장히 중요한 개념이다.
“저장해야 할 데이터와 계산할 수 있는 데이터를 구분하라”

4. 참조를 값으로 바꾸기 vs 값을 참조로 바꾸기

이건 조금 헷갈릴 수 있는데, 핵심은 이거다.

함수에 파라미터로 객체를 넘길지, 아니면 그 안에 있는 값만 넘길지 결정하는 것

예를 들어:

printEmployee(Employee e)

이렇게 객체 자체를 넘길 수도 있고,

printEmployee(String name)

이렇게 객체 안에 있는 값만 넘길 수도 있다.

언제 객체(참조)를 넘기냐?
	•	그 함수가 객체의 여러 데이터가 필요할 때
	•	그 객체 자체가 중요한 개념일 때

언제 값만 넘기냐?
	•	그 함수가 특정 값 하나만 필요할 때
	•	객체에 대한 의존성을 줄이고 싶을 때
	•	재사용성을 높이고 싶을 때

즉 정리하면:

상황	선택
여러 필드 필요	객체(참조) 전달
값 하나만 필요	값 전달
객체 의존 줄이고 싶음	값 전달
객체 자체가 의미 있음	객체 전달

이건 call-by-value, call-by-reference 얘기가 아니라
**“어떤 데이터를 넘기는 게 더 좋은 설계인가?”**에 대한 리팩토링이다.

조건부 로직 간소화
---
1. 조건문 분해하기 (Decompose Conditional)

if문 안에 이런 코드가 있으면:

if (날짜가 말일이고 && VIP고 && 주문금액이 100만원 이상이면) {
    ... 복잡한 계산 ...
} else {
    ... 다른 계산 ...
}

이건 읽기가 너무 어렵다. 그래서 이렇게 바꾼다.

if (isEndOfMonthAndVipAndOverAmount()) {
    calculateDiscount();
} else {
    calculateNormalPrice();
}

즉,
	•	조건 → 함수로 추출
	•	실행 코드 → 함수로 추출

그러면 코드가 아니라 문장처럼 읽힌다.
이게 조건문 분해하기다.

2. 조건문 통합하기

여러 조건을 따로따로 검사하는데 항상 같이 등장하면, 하나로 묶는다.

if (age < 18) return false;
if (age > 65) return false;
if (isSick) return false;

->

if (isNotEligible()) return false;

즉,

여러 조건이 항상 같이 등장하면 하나의 함수로 묶고 이름을 준다.

조건에 이름을 붙이는 리팩토링이다.

3. 중첩 조건문 → 가드 절 (Guard Clause)

이게 굉장히 중요한 개념이다.

나쁜 예 (중첩 if)

if (user != null) {
    if (user.isActive()) {
        if (user.hasPermission()) {
            doSomething();
        }
    }
}

이건 계속 오른쪽으로 밀려서 읽기 어렵다.

가드 절 사용

if (user == null) return;
if (!user.isActive()) return;
if (!user.hasPermission()) return;

doSomething();

이걸 Guard Clause (보호 구문) 라고 한다.
“조건을 만족하지 않으면 바로 종료해서 아래의 정상 흐름을 보호한다”는 의미다.

여기서 중요한 기준이 하나 있다.

상황	사용
if / else 둘 다 중요	if-else
정상 흐름 1개 + 예외 처리 여러 개	Guard Clause

즉,

예외 케이스를 먼저 걸러내고, 정상 흐름만 아래에 남기는 것
이게 Guard Clause다.

실무에서 엄청 많이 쓴다.

4. 조건부 로직을 다형성으로 바꾸기

이건 switch문, if문이 계속 반복될 때 사용하는 방법이다.

예를 들어:

if (type == "VIP") ...
else if (type == "GOLD") ...
else if (type == "NORMAL") ...

이걸 클래스 상속으로 바꾼다.

class Customer {
    int getPoint() {}
}

class VipCustomer extends Customer { ... }
class GoldCustomer extends Customer { ... }
class NormalCustomer extends Customer { ... }

이걸 조건문을 다형성으로 바꾼다고 한다.

즉,

if/switch로 분기하지 말고
객체가 스스로 행동하게 만들어라.

객체지향에서 굉장히 중요한 리팩토링이다.

5. 특수 케이스 추가하기 (Introduce Special Case)

일반적인 로직이 있고, 특정한 경우만 다르게 동작할 때 사용한다.

예:
	•	일반 고객
	•	VIP 고객 (특별 할인)
	•	null 고객 (Null Object)

이걸 if문으로 처리하지 말고 클래스로 분리한다.

특히 여기서 중요한 게 Null Object 패턴이다.

if (customer == null) {
    return 0;
}

이렇게 하지 말고,

class NullCustomer extends Customer {
    int getDiscount() { return 0; }
}

이렇게 하면 if문이 사라진다.

6. Assertion 추가하기

이건 프로그램이 “이 상태는 반드시 참이어야 한다” 라는 것을 코드로 명시하는 것이다.

assert order != null;

이건 에러 처리용이라기보다
개발자에게 알려주는 문서 + 안전장치 같은 것이다.

진짜 핵심만 정리

조건문 리팩토링은 결국 3단계로 생각하면 쉽다.

1단계: 조건문을 함수로 분리 (이름 붙이기)
2단계: Guard Clause로 구조 단순화
3단계: 그래도 복잡하면 다형성으로 제거

이 순서로 리팩토링을 많이 한다.
이건 거의 정석적인 리팩토링 흐름이다.
---

API 리팩토링
---
1. 질의 함수와 변경 함수 분리 (Query vs Modifier)

이건 매우 중요한 원칙이다.
	•	Query: 값만 조회 (상태 변경 없음)
	•	Modifier: 상태를 변경

이 둘을 섞으면 안 된다.

int getTotalPrice() {   // 조회인 줄 알았는데
    totalPrice += 100;  // 값을 바꿔버림
    return totalPrice;
}

이러면 사용하는 사람 입장에서 매우 위험하다.
그래서 원칙은 이것이다.

함수 종류	해야 하는 일
Query	값만 반환
Modifier	상태만 변경
둘 다	하면 안 됨

이걸 Command-Query Separation 원칙이라고 한다.

2. 함수 매개변수화하기 (Parameterize Function)

비슷한 함수가 여러 개 있을 때, 매개변수 하나로 합칠 수 있다.

getDiscountForVip()
getDiscountForGold()

->

getDiscount(CustomerType type)

즉,

함수 여러 개를 만들지 말고, 매개변수로 동작을 바꿔라.

3. 플래그 인수 제거하기

이건 boolean 파라미터 같은 거 말한다.

calculatePrice(order, true);  // true가 뭐야?

이건 API가 나쁜 것이다. 그래서 이렇게 바꾼다.

calculateDiscountPrice(order);
calculateNormalPrice(order);

즉,

boolean 파라미터로 동작을 바꾸지 말고, 함수를 분리하라.

4. 객체 통째로 넘기기 (Preserve Whole Object)

이건 매개변수가 너무 많을 때 사용한다.

printOrder(customerName, customerAddress, customerPhone)

->

printOrder(Customer customer)

즉,

값 여러 개 넘기지 말고 객체 하나로 넘겨라.

5. 매개변수를 질의 함수로 바꾸기

어떤 값이 다른 값으로부터 계산 가능한 경우, 굳이 파라미터로 넘기지 말라는 것이다.

printPrice(order, totalPrice);

->

printPrice(order);  // totalPrice는 order에서 계산

즉,

함수가 스스로 얻을 수 있는 값은 파라미터로 넘기지 말자.

반대도 있다.
질의 함수를 매개변수로 바꾸기 → 함수 내부 의존성을 줄이고 싶을 때 사용.

6. Setter 제거하기

Setter가 많다는 건 객체가 여기저기에서 막 바뀐다는 뜻이다.
가능하면 객체는 immutable(불변) 하게 만드는 게 좋다.

즉,

꼭 필요한 경우가 아니면 setter를 만들지 말자.

7. 생성자를 팩토리 메서드로 바꾸기

이건 꽤 중요한 리팩토링이다.

생성자 방식

new Order("ONLINE");

팩토리 메서드 방식

Order.createOnlineOrder();

팩토리 메서드의 장점:
	•	이름을 붙일 수 있다
	•	어떤 객체를 반환할지 숨길 수 있다
	•	하위 클래스 객체를 반환할 수도 있다

즉,

객체 생성 로직을 숨기고 싶을 때 사용한다.

8. 함수를 명령(Command) 객체로 바꾸기

이건 함수 하나를 클래스로 만드는 것이다. (Command Pattern)

왜 이렇게 하냐면:
	•	undo 기능
	•	실행 취소
	•	실행 기록 저장
	•	작업 큐
	•	다양한 실행 전략

이런 게 필요할 때는 함수 하나로는 부족하고 객체가 필요하다.

상황	선택
단순 작업	함수
상태 필요	클래스
undo 필요	Command 객체
실행을 객체처럼 다뤄야 함	Command 객체

API 설계할 때 항상 이걸 기준으로 보면 된다.

1. 함수 이름만 봐도 무슨 일 하는지 알아야 한다
2. 파라미터는 적을수록 좋다
3. Query는 상태를 바꾸면 안 된다
4. 객체 생성은 숨기는 게 좋다
5. 단순하면 함수, 복잡하면 객체(Command)

이 카탈로그는 결국 **“좋은 함수 형태 만들기”**에 대한 리팩토링이라고 보면 된다.

상속 다루기
---
1. 메서드/필드 올리기 (Pull Up)

여러 서브클래스에서 같은 코드가 있으면 부모 클래스로 올린다.

class Dog {
    void eat() { ... }
}

class Cat {
    void eat() { ... }
}

→

class Animal {
    void eat() { ... }
}

class Dog extends Animal {}
class Cat extends Animal {}

기준은 간단하다.

여러 자식이 공통으로 사용 → 위로 올린다

⸻

2. 메서드/필드 내리기 (Push Down)

특정 서브클래스에서만 사용하는 기능이면 아래로 내린다.

class Animal {
    void fly() { ... }  // 모든 동물이 날지 않음
}

->

class Animal {}

class Bird extends Animal {
    void fly() { ... }
}

기준:

일부 자식만 사용 -> 아래로 내린다

3. 생성자 본문 올리기

서브클래스 생성자에서 중복되는 코드가 있으면 부모 생성자로 올린다.

class Dog {
    Dog() {
        this.age = 0;
    }
}

class Cat {
    Cat() {
        this.age = 0;
    }
}

->

class Animal {
    Animal() {
        this.age = 0;
    }
}


4. 타입 코드를 서브클래스로 바꾸기

이건 중요한 리팩토링이다.

before
class Employee {
    String type; // "FULL_TIME", "PART_TIME"

    int getPay() {
        if (type.equals("FULL_TIME")) return 100;
        else return 50;
    }
}

after
class Employee {
    int getPay() { return 0; }
}

class FullTimeEmployee extends Employee {
    int getPay() { return 100; }
}

class PartTimeEmployee extends Employee {
    int getPay() { return 50; }
}

즉,

type + if문 → 상속 + 오버라이딩

이건 조건문을 다형성으로 바꾸기와 같은 개념이다.

5. 서브클래스 제거하기

리팩토링을 계속 하다 보면, 어떤 서브클래스는 하는 일이 없어질 수 있다.
그러면 그 서브클래스는 삭제한다.

즉,

서브클래스가 아무 일도 안 하면 삭제

6. 슈퍼클래스 추출하기

상속 구조가 아예 없는데, 여러 클래스에 공통 기능이 있으면 부모 클래스를 새로 만든다.

class Dog { void eat() {} }
class Cat { void eat() {} }

->

class Animal { void eat() {} }
class Dog extends Animal {}
class Cat extends Animal {}


⸻

7. 계층 합치기 (Collapse Hierarchy)

부모 클래스와 자식 클래스가 거의 차이가 없으면, 상속 구조를 없애고 하나로 합친다.

즉,

상속 구조가 의미 없으면 없애라

⸻

8. 상속 → 위임으로 바꾸기 (중요)

이게 진짜 중요하다. 실무에서 많이 쓴다.

상속은 결합도가 매우 강하다. (부모 바뀌면 자식 다 영향 받음)
그래서 상속 대신 위임(Delegation) 을 사용하기도 한다.

상속

class Engine {
    void start() {}
}

class Car extends Engine {
}

위임

class Engine {
    void start() {}
}

class Car {
    private Engine engine = new Engine();

    void start() {
        engine.start();
    }
}

차이:

상속	위임
강한 결합	느슨한 결합
컴파일 타임	런타임 교체 가능
부모 변경 영향 큼	영향 적음
구조 고정	유연함

그래서 객체지향에서 유명한 말이 있다.

상속보다 조합(위임)을 더 선호하라
(Favor composition over inheritance)

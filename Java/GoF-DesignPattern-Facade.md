### 1. 개념: 건물의 정면 (Facade)
* **어원:** 프랑스어로 건물의 **'정면(외벽)'**을 뜻함.
* **의미:** 건물 밖에서는 멋진 입구만 보이고 내부의 복잡한 배관, 전기 배선 등은 보이지 않는 것처럼, 소프트웨어에서도 **복잡한 서브 시스템(라이브러리, 프레임워크 등)을 뒤로 숨기고, 클라이언트에게는 단순한 인터페이스(창구)를 제공**하는 패턴.

### 2. 예시: 자바 이메일 발송 기능
* **문제 상황 (퍼사드 적용 전):**
    * 이메일을 보내려면 `Properties`, `Session`, `MimeMessage`, `Transport` 등 수많은 로우 레벨(Low-level) 객체들을 클라이언트가 직접 생성하고 조작해야 함.
    * **결과:** 클라이언트 코드가 복잡해지고, 특정 라이브러리에 강하게 결합(Tight Coupling)되며, 테스트하기 어려워짐.
* **해결책 (퍼사드 적용 후):**
    * `EmailSender`(퍼사드)라는 클래스를 만들고, 복잡한 로직을 그 안에 다 숨김.
    * 클라이언트는 `emailSender.sendEmail(to, subject, text)` 처럼 단순한 메서드 하나만 호출하면 됨.

### 3. 장단점
* **장점:**
    * **느슨한 결합 (Loose Coupling):** 클라이언트가 복잡한 서브 시스템을 몰라도 되므로 의존성이 줄어듦.
    * **가독성:** 코드가 간결하고 읽기 쉬워짐.
    * **테스트 용이:** 복잡한 `static` 메서드 등을 직접 호출하는 대신, 퍼사드 객체만 Mocking 하여 테스트하기 쉬워짐.
* **단점:**
    * **의존성의 이동:** 클라이언트의 짐을 덜어준 것일 뿐, 퍼사드 클래스 자체는 여전히 서브 시스템의 모든 복잡한 의존성을 짊어지고 있음.

### 4. 실무 활용 사례 (Spring Framework)
"스프링의 많은 부분이 일종의 퍼사드 패턴"이라고 설명함.

* **`MailSender`:** 복잡한 `JavaMail` API를 감싸서 단순하게 메일을 보낼 수 있게 해줌.
* **`PlatformTransactionManager`:** JDBC, JPA, Hibernate 등 각기 다른 트랜잭션 기술을 추상화하여, 클라이언트는 구체적인 기술을 몰라도 트랜잭션을 처리할 수 있게 함.
* **Spring MVC:** 서블릿(Servlet) 기술을 뒤로 숨기고, 애노테이션(`@Controller`, `@GetMapping`) 기반의 편리한 인터페이스를 제공함. (심지어 WebFlux로 기술이 바뀌어도 클라이언트 코드는 유지됨).

### 5. 브리지 패턴 vs 퍼사드 패턴
* 구조적으로는 인터페이스 뒤에 구현을 숨긴다는 점에서 비슷해 보일 수 있음.
* **차이점 (의도):**
    * **브리지 패턴:** 추상층과 구현층을 분리하여 **서로 독립적으로 확장**하는 것이 목적.
    * **퍼사드 패턴:** 복잡한 서브 시스템을 **단순하게 보여주고(인터페이스 제공), 기술을 은닉**하는 것이 목적.

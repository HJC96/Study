### 1. 템플릿 메서드 패턴 (Template Method Pattern)
* **개념:** 알고리즘의 **구조(뼈대)**는 상위 클래스(Abstract Class)의 메서드(Template Method)에 정의하고, 구체적인 **단계(Step)**의 구현은 하위 클래스(Concrete Class)에게 위임하는 패턴임.
* **구조:**
    * **Abstract Class:** `process()` 같은 템플릿 메서드 안에 전체 흐름(로직)을 정의함. 변경이 필요한 부분은 추상 메서드(`getResult()`)로 선언.
    * **Concrete Class:** 추상 메서드를 상속받아 구체적인 로직(예: 덧셈, 곱셈)을 구현함.
* **핵심 원칙:**
    * **IoC (제어의 역전):** 하위 클래스의 코드를 상위 클래스의 템플릿 메서드가 호출함.
    * **코드 중복 제거:** 공통 로직(파일 읽기, 예외 처리 등)은 상위 클래스에 한 번만 작성함.

### 2. 템플릿 콜백 패턴 (Template Callback Pattern)
* **개념:** 상속 대신 **위임(Delegation)**을 사용하는 방식. 전략 패턴의 변형으로, 실행 로직(Callback)을 메서드 파라미터로 전달함.
* **특징:**
    * **상속 불필요:** `extends` 없이 인터페이스(`Callback`)를 구현하여 사용.
    * **유연성:** 익명 내부 클래스나 람다(Lambda)를 사용하여 코드를 간결하게 작성 가능.
    * 스프링에서 `JdbcTemplate`, `RestTemplate` 등이 이 패턴을 사용함.

### 3. 장단점 (템플릿 메서드 패턴)
* **장점:**
    * **코드 재사용:** 중복 코드를 상위 클래스로 모아 관리하기 쉬움.
    * **확장성 (OCP):** 기존 템플릿 코드를 건드리지 않고 새로운 기능(곱셈 등)을 추가할 수 있음.
* **단점:**
    * **리스코프 치환 원칙(LSP) 위반 가능성:** 상속을 사용하므로, 하위 클래스가 상위 클래스의 의도를 벗어난 동작을 할 위험이 있음.
    * **복잡도 증가:** 알고리즘 구조가 복잡해지면 템플릿 유지보수가 어려워질 수 있음.

### 4. 실무 활용 사례 (Java & Spring)
* **Java - `HttpServlet`:**
    * `service()` 메서드(템플릿)가 내부적으로 `doGet()`, `doPost()` 등을 호출함. 개발자는 필요한 메서드만 오버라이딩하면 됨.
* **Spring - `Configuration`:**
    * `WebSecurityConfigurerAdapter` 등의 설정 클래스를 상속받아 필요한 부분(`init` 등)만 구현하면, 스프링이 전체 설정 흐름 안에서 이를 호출함.
* **Spring - `JdbcTemplate`, `RestTemplate`:**
    * **템플릿 콜백 패턴**의 대표적인 예.
    * `execute()`, `query()` 등의 메서드(템플릿)가 리소스 관리(Connection 생성/종료, 예외 처리)를 담당하고, 실제 실행할 SQL이나 로직은 콜백(Callback)으로 받아서 처리함.
